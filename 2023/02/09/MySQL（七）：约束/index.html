数据完整性与约束的分类

  • 为什么需要约束？

    为了保证数据的完整性

  • 约束：对表中字段的限制

  • 约束的分类：

    • 角度一：约束字段的个数
      • 单列约束 vs 多列约束
    • 角度二：约束的作用范围
      • 列级约束：将此约束声明在对应字段的后面
      • 表级约束：在表中所有字段都声明完，在所有字段的后面声明的约束
    • 角度三：约束的作用（功能）
      • not null (非空约束)
      • unique （唯一性约束）
      • primary key (主键约束)
      • foreign key (外键约束)
      • check （检查约束）
      • defualt (默认值约束)

  • 如何添加/删除约束？

    • CREATE TABLE
    • ALTER TABLE

  • 如何查看表中的约束？

非空约束（not null）

  • 作用：限定某个字段/某列的值不允许为空

唯一性约束（unique）

  • 作用：限制某个字段/某列的值不能重复
  • 删除唯一性约束
    • 添加唯一性约束上的列也会自动创建唯一索引
    • 删除唯一约束只能通过删除唯一索引的方式删除
    • 删除时需要指定唯一索引名，唯一索引名就和唯一约束名一样
    • 如果创建唯一约束时未指定名称，如果是单列，就默认和列名相同；如果是组合列，那么默认和（）中排在第一个的列名相同。也可以自定义唯一性约束名。

主键约束（primary key）

  • 作用：用来唯一标识表中的一行记录
  • 特点：主键约束相当于“唯一约束+非空约束的组合”，不允许重复、也不允许出现空置。
  • 一个表中最多只能有一个主键约束
  • MySQL的主键名总是 PRIMARY
  • 当创建主键约束时，系统默认会在所在的列或列组合上建立对应的主键索引（能够根据主键查询的，就根据主键查询，效率更高）。如果删除主键约束了，主键约束对应的索引就自动删除了。

自增长列：AUTO_INCREMENT

  • 作用：某个字段的值自增
  • 一个表最多只能由一个自增列
  • 自增长列约束的列必须是键列（主键列，唯一链）
  • 自增约束的列数据类型必须时整数类型
  • 当向主键（含AUTO_INCREMENT）的字段上添加0或null，实际上会自动的往上添加
  • 开发中，一旦主键字段上声明有AUTO_INCREMENT，则添加数据时就不要给主键赋值

外键（foreign key）约束

  • 作用：限定某个表的某个字段的引用完整性
  • 特点：
    • 一个表中可以有多个外键约束
    • 从表（子表）的外键列，必须引用/参考主表（父表）的主键或唯一约束的列。
    • 在创建外键约束时，如果不给外键约束命名，默认名不是列明，而是自动产生一个外键名，也可以自定义名。
    • 创建表时，先创建主表再创建从表
    • 删除表时，先删从表再删主表
    • 当创建外键约束时，系统默认会在所在的列上建立对应的普通索引。
    • 删除外键约束后，必须手动删除对应的索引。
    • 外键约束有成本，需要消耗系统资源，对于大并发的SQL不合适，可能会因为外键约束的系统开销而变得非常慢。（实际开发中，建议在应用层面解决外键概念）

约束等级

  • Cascade方式：在父表上update/delete记录时，同步update/delete掉子表的匹配记录
  • Set null方式：在父表上update/delete记录时，将子表的匹配记录的列设为null，但是要注意子表的外键列不能未not null
  • No action方式：如果子表中有匹配的记录，则不允许对附表对应候选键进行update/delete操作
  • Restrict方式：同 No action
  • Set default方式：父表有变更时，子表将外键列设置成一个默认的值，但Innodb不能识别

对于外键约束，最好采用ON UPDATE CASCADE ON DELETE SET NULL

检查约束（check）

  • 作用：检查某个字段的值是否符合xx要求，一般值的是值的范围

默认值约束（defualt)

  • 作用：给某个字段指定默认值

面试题：

  • 为什么不想要null值？
    • 不好比较。null碰到运算符通常返回null
    • 效率不高。影响提高索引效果，因此建表时通常not null default ‘’ 或 default 0
